# About

I build software for robots that need to know where they are. My focus is sensor fusion — combining noisy signals from IMUs, GPS, and wheel encoders into a single state estimate that's actually good enough to navigate from.

I'm the author of FusionCore, a ROS 2 UKF I started after hitting the same walls with robot_localization too many times. I've also done hands-on hardware bring-up on real mobile platforms (Clearpath Husky), integrated full Nav2 navigation stacks, and run reproducible, honest benchmarks across six outdoor sequences spanning all four seasons.

I care about correctness over cleverness — and about shipping things that actually work on real robots, not just in simulation.

FusionCore

ROS 2 UKF sensor fusion  ·  C++17  ·  Apache 2.0  ·  Active

A 22-state Unscented Kalman Filter for mobile robots. Fuses IMU, wheel encoders, and GPS into one position estimate at 100 Hz. Estimates IMU bias online. Adapts noise covariance from the innovation sequence automatically — no hand-tuning.

## Why it exists

robot_localization is the default ROS answer. It works, but it has real friction: GPS goes through navsat_transform (extra node, added latency, breaks at UTM zone boundaries), no IMU bias estimation, and covariances you tune by hand and retune every time your hardware or environment changes. FusionCore handles ECEF GPS natively — no projection, no extra node. Chi-squared gating on every sensor.

Husky + BNO085 Hardware Bringup

Clearpath Husky UGV  ·  BNO085 IMU  ·  ROS 2 Jazzy

End-to-end hardware bring-up for a Clearpath Husky UGV running FusionCore on ROS 2 Jazzy, using the BNO085 IMU. This involved writing hardware-specific sensor configs, calibrating IMU-to-base-link transforms, and tuning the EKF process noise for the Husky's ground-contact dynamics.

## What it covers

BNO085 → ROS 2 integration

Driver configuration, quaternion output validation, gravity subtraction verification. Confirmed IMU data quality before passing into the fusion stack.

robot_localization migration guide

Step-by-step guide for teams migrating from robot_localization EKF to FusionCore on new platforms — what parameters map to what, what to discard, what to re-tune.

Environment preset YAMLs

GPS quality scenario presets: open sky, urban canyon, degraded signal. Drop-in config files so teams don't have to retune from scratch per deployment site.

~/portfolio/PROJECTS/nav2_bridge.cpp

Nav2 Integration

FusionCore as Nav2 localization source  ·  Full navigation stack

Wired FusionCore into a full Nav2 navigation stack. FusionCore publishes the map → odom → base_link TF chain and an /odom topic that Nav2's controller and planner consume directly.

## What was involved

TF chain management

FusionCore owns the odom → base_link transform. A static map → odom identity bridge handles the Nav2 localization assumption. No AMCL, no slam_toolbox required for GPS-primary environments.

Launch architecture

fusioncore_nav2.launch.py — single launch file that brings up the full stack: FusionCore, navsat_transform (optional), Nav2 BT navigator, costmap, and the controller server.

Outdoor navigation

Validated on outdoor campus routes. GPS provides global position; wheel odometry + IMU provide smooth inter-GPS interpolation. Nav2 receives a 100 Hz pose stream with correct covariance.

NCLT Benchmark Pipeline

Reproducible  ·  Open methodology  ·  RTK GPS ground truth

A fully reproducible pipeline for benchmarking mobile robot localization on the NCLT dataset — six sequences across all four seasons, RTK GPS as ground truth.

## Design principles

Most sensor fusion benchmarks report only their best sequences. This one reports all six, including the one FusionCore loses. The RL comparison uses Mahalanobis gating enabled (chi²(2, 0.999) = 13.82) — the fairest possible comparison.

Pipeline

ROS 2 bag playback at 3× speed
→ FusionCore + RL-EKF run simultaneously
→ Trajectories extracted to TUM format
→ evo_ape ATE evaluation with SE3 alignment
→ Publication-quality matplotlib figures generated
